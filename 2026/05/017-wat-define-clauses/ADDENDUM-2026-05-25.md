## RENAME (LOCKED 2026-05-25)

```
:wat::core::define-clauses  →  :wat::core::defclause
```

Single noun. Clojure-resonant (matches `defn` shape). Per arc 109's directional push toward Clojure (vs Scheme paren-density).

The boundary:
- `:wat::core::defn` — single-arity functions WITHOUT guards or post-conditions. The common case.
- `:wat::core::defclause` — multi-arity OR guard-needing OR post-condition-needing OR both. Subsumes Clojure N-ary + Erlang clause-selection + Eiffel DbC.

A single-clause `defclause` with no guard is essentially a `defn` (legal but verbose for that case; user picks the form matching intent).

## CLAUSE-LEVEL KEYWORDS (LOCKED 2026-05-25 via intueri cast)

```
:guard  — clause-selection expression (formerly :when)
:ensure — output-validation explicit :fn (formerly :post)
```

Both names selected via intueri cast. Pair-level argument:
- Shared semantic register (formal-correctness vocabulary; guard theory + Design by Contract)
- Independent of language heritage (neither raw Erlang nor raw Clojure transplant)
- Temporal ordering INSIDE the words ("guard the entry; ensure the exit")
- Error behavior INSIDE the words (guard fails → clause steps aside; ensure fails → raises)
- Both pass all four intueri questions atomically

Intueri's strongest argument was the rejected-pair analysis:
- `:when` failed Honest under arc 109's Clojure-push (Clojure `:when` = `if-then-nil`, not "clause guard with fallthrough"); would create silent semantic fork
- `:post` failed Obvious (3-letter borrowed prefix requiring Clojure knowledge) + Honest (shape mismatch with Clojure's implicit `%`)

Both `:guard` + `:ensure` pass 4/4 intueri questions. Recommendation: lock the pair.

## CLAUSE SHAPE (LOCKED)

Per clause:
```
(args :guard expr :ensure :fn body)
```

Where:
- `args` = vector `[name <- :Type  name <- :Type ...]` (mandatory; Clojure-style vector + wat `<-` arrow per arc 234 convention)
- `:guard expr` = optional; single boolean expression in clause-arg scope; false → try next clause
- `:ensure :fn` = optional; explicit `:fn` requiring new binding for return value; false → raises `:PostconditionFailed`
- `body` = mandatory; the clause's evaluation expression

Minimal clause: `(args body)` — no guard, no ensure.

## LITERAL PATTERNS — NOT SUPPORTED (LOCKED 2026-05-25)

Path C selected over Path A (`[0 <- :wat::core::i64]`) and Path B (`[0]` inferred).

**Rationale:** the arg-binding contract from arc 159 + arc 169 + arc 234 is sacred. Arg position is ALWAYS a binding-name-with-explicit-type. Breaking this contract for defclause sugar creates a special case the substrate (and readers) must learn.

Literal-matches become guards instead:

```wat
;; instead of [0 <- :wat::core::i64] literal-pattern
([n <- :wat::core::i64]
 :guard (:wat::core::i64::= n 0)
 1)
```

Verbose-but-honest per `feedback_verbose_is_honest`. One mechanism (guards) handles ALL clause-discrimination logic.

Clojure-resonance favors Path C (Clojure's `defn` doesn't have arg-position literal patterns).

## DISPATCH SEMANTICS (LOCKED)

- **First-match-wins** — clauses tried in declaration order; first matching clause (arity match + guard true) fires
- **User controls priority by clause order** — no implicit ordering rules the substrate enforces
- **All clauses fall through → `:NoMatchingClause` error** with declared clauses + their guards listed
- **`:ensure` raises hard error on false** (does NOT try next clause)

---

## DEFCLAUSE-EXCLUSIVE FEATURES (LOCKED)

- `:guard` and `:ensure` are defclause-exclusive
- `:wat::core::defn` stays MINIMAL — single-arity, no guards, no post-conditions
- If user wants clauses → use defclause (the form name signals the capability)

## CANONICAL DEMOS (LOCKED 2026-05-25)

### Demo 1 — Factorial (Erlang's spirit via Path C)

```wat
(:wat::core::defclause :my::factorial -> :wat::core::i64

  ;; clause 1: base case (n = 0)
  ([n <- :wat::core::i64]
   :guard (:wat::core::i64::= n 0)
   1)

  ;; clause 2: recursive case (n > 0)
  ([n <- :wat::core::i64]
   :guard (:wat::core::i64::> n 0)
   (:wat::core::i64::* n (:my::factorial (:wat::core::i64::- n 1))))

  ;; clause 3: negative case (n < 0)
  ([n <- :wat::core::i64]
   :guard (:wat::core::i64::< n 0)
   (:wat::core::error :NegativeFactorial n)))
```

Dispatch:
- `(:my::factorial 5)` → clause 2; recurses through factorial(4), factorial(3), ..., factorial(0) → clause 1; returns 120
- `(:my::factorial 0)` → clause 1; returns 1
- `(:my::factorial -3)` → clause 3; raises `:NegativeFactorial -3`

### Demo 2 — Complex (2 same-arity guards + 3-arity with ensure)

```wat
(:wat::core::defclause :my::process -> :wat::core::String

  ;; clause 1: 2-arity; selects when x > y
  ([x <- :wat::core::i64  y <- :wat::core::i64]
   :guard (:wat::core::i64::> x y)
   (:wat::core::String::concat "x>y: " (:wat::core::i64/to-string x)))

  ;; clause 2: 2-arity (SAME arity); selects when x < y
  ([x <- :wat::core::i64  y <- :wat::core::i64]
   :guard (:wat::core::i64::< x y)
   (:wat::core::String::concat "x<y: " (:wat::core::i64/to-string y)))

  ;; clause 3: 3-arity; no guard; ensures output starts with "result:"
  ([x <- :wat::core::i64  y <- :wat::core::i64  z <- :wat::core::i64]
   :ensure (:wat::core::fn [result <- :wat::core::String] -> :wat::core::bool
             (:wat::core::String/starts-with? result "result:"))
   (:wat::core::String::concat "result: sum="
     (:wat::core::i64/to-string
       (:wat::core::i64::+ (:wat::core::i64::+ x y) z)))))
```

Dispatch:
- `(:my::process 5 3)` → 2-arity; clause 1 guard `5 > 3` → true; returns `"x>y: 5"`
- `(:my::process 3 5)` → 2-arity; clause 1 false → try clause 2; guard `3 < 5` → true; returns `"x<y: 5"`
- `(:my::process 3 3)` → 2-arity; clause 1 false; clause 2 false; no more 2-arity clauses + 3-arity clause requires 3 args; `:NoMatchingClause` error
- `(:my::process 1 2 3)` → 3-arity; clause 3; body returns `"result: sum=6"`; `:ensure` validates starts-with? `"result:"` → true; returns body's result
- (If clause 3's body were modified to not start with "result:", `:ensure` raises `:PostconditionFailed`)
